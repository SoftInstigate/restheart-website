Team invitations allow users with the `owner` role to add people to their tenant.

[IMPORTANT]
====
The invite endpoint requires the JWT to contain the active tenant claim. Ensure the following configuration:

* `accountsConfig.tenant-claim-name: tenant` (or your custom claim name)
* `jwtTokenManager.account-properties-claims: [tenant]`
* `accountsConfig.account-properties-claims: [tenant]`

Without these, the service cannot determine the caller's team role and returns `403 Forbidden`.
====
The behavior differs depending on whether the invited user already has an account.

[source,mermaid]
----
flowchart TD
  A[POST /auth/invite\nrequires owner role] --> B{User exists?}
  B -- No --> C[Create user\nroles: $unauthenticated]
  C --> D[MembershipProvider.addMember\nuserId, tenantId, role]
  D --> E[Email: /auth/activate\n?email=â€¦&token=â€¦]
  E --> F[PATCH /auth/activate\nemail + token + password]
  F --> G[Set password, roles â†’ user\nAuto-login JWT cookie]

  B -- Yes, not in tenant --> H[Create invitation\nin auth_invitations\nTTL 7 days]
  H --> I[Email: /invitations/accept\n?token=â€¦]
  I --> J[POST /auth/accept-invite\ntoken â€” auth required]
  J --> K[MembershipProvider.addMember\nDelete invitation doc\n200 OK]

  B -- Yes, already member --> L[409 Conflict]
----

== Invite a user

`POST /auth/invite` â€” requires role `owner`.

=== Request

[source,http]
----
POST /auth/invite
Authorization: Bearer <token>
Content-Type: application/json

{
  "email": "bob@example.com",
  "role":  "member"
}
----

`role` must be one of: the configured `member-role-name` (default `"member"`) or the configured `ownership-role` (default `"owner"`).

IMPORTANT: The `admin` role is a system-level ACL role in RESTHeart and should NEVER be used as a team/membership role. Using `admin` as a membership role grants the user unrestricted access to the entire system.

=== Server-side steps

. Authorize the caller (must be `owner` of their tenant).
. Validate `email` format and `role` value.
. **If the user already belongs to the caller's tenant** â†’ `409 Conflict`.
. **If the user does not exist** â†’ create user with `roles: ["$unauthenticated"]`, add membership via `MembershipProvider.addMember()`, send activation email with link to `{frontend-url}/auth/activate?email=...&token=...`.
. **If the user exists but is NOT in this tenant** â†’ create an invitation document in the `auth_invitations` collection with `{ email, token, orgId, role, createdAt, expiresAt }`. Do NOT add membership yet. Send invite email with link to `{frontend-url}/invitations/accept?token=...`.
. Return `201 Created`.

[NOTE]
====
Invitations for existing users are stored in the `auth_invitations` collection (same database as `users` and `acl`), not on the user document. This allows multiple pending invitations per user across different teams.

The `auth_invitations` document schema:

[source,json]
----
{
  "_id": {"$oid": "..."},
  "email": "user@example.com",
  "token": "64-char hex",
  "orgId": {"$oid": "..."},
  "role": "member",
  "createdAt": {"$date": 1234567890},
  "expiresAt": {"$date": 1234567890}
}
----
====

== Activate an invitation (new user)

`PATCH /auth/activate` â€” public (the user has only the token link).

=== Request

[source,http]
----
PATCH /auth/activate
Content-Type: application/json

{
  "email":    "bob@example.com",
  "token":    "<token from email>",
  "password": "correct-horse-battery"
}
----

=== Server-side steps

. Find user by `email`.
. Compare `inviteToken` using *constant-time comparison*.
. Check `inviteCreatedAt` + TTL (7 days).
. Enforce password strength.
. Set password, assign `roles: ["user"]`.
. `$unset` invite token and creation timestamp.
. Issue JWT + refresh cookie â†’ auto-login.
. Return `200 OK`.

[NOTE]
====
Consents are not managed by this endpoint.
Use a response interceptor on the deployment layer to persist consents from the request body.
See the `AccountsConsentsSaver` interceptor in `restheart-cloud-server` for a reference implementation.
====

=== Error responses

|===
| Status | Reason

| `400` | Missing fields; password too weak
| `401` | Invalid or expired invite token
|===

== Accept an invitation (existing user)

`POST /auth/accept-invite` â€” requires authentication.

=== Request

[source,http]
----
POST /auth/accept-invite
Authorization: Bearer <token>
Content-Type: application/json

{
  "token": "<token from email>"
}
----

=== Server-side steps

. Find user by authenticated email.
. Verify invite token matches and is not expired (7 days).
. `MembershipProvider.addMember(userId, orgId, role)` â€” adds user to team.
. Clear invite fields (`inviteToken`, `inviteCreatedAt`, `inviteOrgId`, `inviteRole`).
. Return `200 OK`.

=== Frontend flow

. User clicks link in email â†’ `/invitations/accept?token=...`
. If not logged in â†’ component shows "Sign in to accept" with a login button that preserves the return URL
. After login â†’ component calls `POST /auth/accept-invite { token }`
. On success â†’ redirect to dashboard

== Accept via OAuth (new user)

The activation page (`/auth/activate`) shows OAuth buttons with the `pendingInviteToken` passed to the OAuth authorize URL.

=== OAuth URL

[source]
----
{api-base-url}/auth/oauth/authorize/{provider}?pendingInviteToken=...
----

=== Flow

. User clicks "Sign in with Google/GitHub" on the activation page.
. OAuth callback detects `$unauthenticated` user.
. `MembershipProvider.activateViaOAuth(userId, consents)` â€” assigns `roles: ["user"]`, stores consents.
. JWT cookie set, redirect to `frontend-app-url`.

[NOTE]
====
If the OAuth email does not match the invited email, the callback redirects to `{frontend-error-url}?reason=The+email+used+for+social+login+does+not+match+the+invited+email`.
====

== Re-send an invitation

`POST /auth/resend-invite` â€” requires role `owner`.

*Request body:*

[source,json]
----
{ "email": "bob@example.com" }
----

Generates a new `inviteToken` (invalidating the previous one) and re-sends the invitation email.
Returns `200 OK` on success.

== Remove a member

`DELETE /auth/remove-member` â€” requires role `owner`.

=== Request

[source,http]
----
DELETE /auth/remove-member
Authorization: Bearer <token>
Content-Type: application/json

{
  "email": "bob@example.com"
}
----

=== Server-side steps

. Authorize the caller (must be `owner` of their active tenant).
. Verify the target user is a member of the caller's tenant â€” `404 Not Found` otherwise.
. Prevent the owner from removing themselves (`400 Bad Request`).
. Call `MembershipProvider.removeMember(email, tenantId)`.
. Return `200 OK`.

=== Error responses

|===
| Status | Reason

| `400` | Missing `email` field; or owner trying to remove themselves
| `403` | Caller does not have `owner` role
| `404` | Target user is not a member of the caller's tenant
|===

== Update a member's role

`PATCH /auth/member-role` â€” requires role `owner`.

=== Request

[source,http]
----
PATCH /auth/member-role
Authorization: Bearer <token>
Content-Type: application/json

{
  "email": "bob@example.com",
  "role":  "member"
}
----

`role` must be one of: the configured `member-role-name` (default `"member"`) or the configured `ownership-role` (default `"owner"`).

IMPORTANT: The `admin` role is a system-level ACL role and cannot be assigned as a team role via this endpoint.

=== Server-side steps

. Authorize the caller (must be `owner` of their active tenant).
. Validate `role` â€” `400 Bad Request` if not `memberRoleName` or `ownershipRole`.
. Verify the target user is a member of the caller's tenant â€” `404 Not Found` otherwise.
. Call `MembershipProvider.updateMemberRole(email, tenantId, newRole)`.
. Return `200 OK`.

=== Error responses

|===
| Status | Reason

| `400` | Missing fields; or `role` not in `[memberRoleName, ownershipRole]`
| `403` | Caller does not have `owner` role
| `404` | Target user is not a member of the caller's tenant
|===
