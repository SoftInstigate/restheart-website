Team invitations allow users with the `owner` or `admin` role to add people to their tenant.
The behavior differs depending on whether the invited user already has an account.

== Invite a user

`POST /auth/invite` — requires role `owner` or `admin`.

=== Request

[source,http]
----
POST /auth/invite
Authorization: Bearer <token>
Content-Type: application/json

{
  "email": "bob@example.com",
  "role":  "member"
}
----

`role` must be one of: the configured `member-role-name` (default `"member"`) or `"admin"`.

=== Server-side steps

. Authorize the caller (must be `owner` or `admin` of their tenant).
. Validate `email` format and `role` value.
. **If the user already belongs to the caller's tenant** → `409 Conflict`.
. **If the user does not exist** → create user with `roles: ["$unauthenticated"]`, add membership via `MembershipProvider.addMember()`, send activation email with link to `{frontend-url}/auth/activate?email=...&token=...`.
. **If the user exists but is NOT in this tenant** → store `inviteToken`, `inviteOrgId`, `inviteRole` on the user document (do NOT add membership yet). Send invite email with link to `{frontend-url}/invitations/accept?token=...`.
. Return `201 Created`.

=== Email links

[cols="1,1"]
|===
| User type | Email link

| New user (no account)
| `{frontend-url}/auth/activate?email=...&token=...`

| Existing user
| `{frontend-url}/invitations/accept?token=...`
|===

== Activate an invitation (new user)

`PATCH /auth/activate` — public (the user has only the token link).

=== Request

[source,http]
----
PATCH /auth/activate
Content-Type: application/json

{
  "email":    "bob@example.com",
  "token":    "<token from email>",
  "password": "correct-horse-battery"
}
----

=== Server-side steps

. Find user by `email`.
. Compare `inviteToken` using *constant-time comparison*.
. Check `inviteCreatedAt` + TTL (7 days).
. Enforce password strength.
. Set password, assign `roles: ["user"]`.
. `$unset` invite token and creation timestamp.
. Issue JWT + refresh cookie → auto-login.
. Return `200 OK`.

[NOTE]
====
Consents are not managed by this endpoint.
Use a response interceptor on the deployment layer to persist consents from the request body.
See the `AccountsConsentsSaver` interceptor in `restheart-cloud-server` for a reference implementation.
====

=== Error responses

|===
| Status | Reason

| `400` | Missing fields; password too weak
| `401` | Invalid or expired invite token
|===

== Accept an invitation (existing user)

`POST /auth/accept-invite` — requires authentication.

=== Request

[source,http]
----
POST /auth/accept-invite
Authorization: Bearer <token>
Content-Type: application/json

{
  "token": "<token from email>"
}
----

=== Server-side steps

. Find user by authenticated email.
. Verify invite token matches and is not expired (7 days).
. `MembershipProvider.addMember(userId, orgId, role)` — adds user to team.
. Clear invite fields (`inviteToken`, `inviteCreatedAt`, `inviteOrgId`, `inviteRole`).
. Return `200 OK`.

=== Frontend flow

. User clicks link in email → `/invitations/accept?token=...`
. If not logged in → `AuthGuard` redirects to `/auth/login?returnUrl=/invitations/accept?token=...`
. After login → component calls `POST /auth/accept-invite { token }`
. On success → redirect to dashboard

== Accept via OAuth (new user)

The activation page (`/auth/activate`) shows OAuth buttons with the `pendingInviteToken` passed to the OAuth authorize URL.

=== OAuth URL

[source]
----
{api-base-url}/auth/oauth/authorize/{provider}?pendingInviteToken=...
----

=== Flow

. User clicks "Sign in with Google/GitHub" on the activation page.
. OAuth callback detects `$unauthenticated` user.
. `MembershipProvider.activateViaOAuth(userId, consents)` — assigns `roles: ["user"]`, stores consents.
. JWT cookie set, redirect to `frontend-app-url`.

[NOTE]
====
If the OAuth email does not match the invited email, the callback redirects to `{frontend-error-url}?reason=The+email+used+for+social+login+does+not+match+the+invited+email`.
====

== Re-send an invitation

`POST /auth/resend-invite` — requires role `owner` or `admin`.

*Request body:*

[source,json]
----
{ "email": "bob@example.com" }
----

Generates a new `inviteToken` (invalidating the previous one) and re-sends the invitation email.
Returns `200 OK` on success.
