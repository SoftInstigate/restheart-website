A user can belong to multiple teams simultaneously, each with a different role.

[TIP]
====
*9.6.0 breaking change:* the `team` JWT claim and the `POST /auth/switch-team` response changed from a scalar id to `{ _id, role }`. 
ACL filters using `@user.team` as the team id must migrate to `@user.team._id`. 
`@user.team.role` is now available for predicate-based gating (e.g. `equals[@user.team.role, "owner"]`). 
Clients reading the `team` claim must adapt: `team._id` replaces `team`.
====

The active team and its corresponding role are encoded in the JWT; switching teams reissues the JWT without requiring a new login.

[NOTE]
====
The membership model — field names, role vocabulary, storage collection — can be customised via the xref:membership-providers.adoc[`MembershipProvider` SPI] (9.4.1+).
This page describes the built-in default behaviour.
====

=== Data model

Each user document stores a `teams` array alongside the scalar `team` field.
The `DefaultMembershipProvider` stores team identifiers as MongoDB `ObjectId` values:

[source,json]
----
{
  "_id":     "alice@example.com",
  "roles":   ["user"],                                        // system ACL role (from default-role)
  "team": { "_id": { "$oid": "64a1b2c3d4e5f6a7b8c9d0e1" }, "role": "owner" },  // active team { _id, role } (JWT claim)
  "teams": [
    { "id": { "$oid": "64a1b2c3d4e5f6a7b8c9d0e1" }, "role": "owner"  },  // membership role
    { "id": { "$oid": "64a1b2c3d4e5f6a7b8c9d0e2" }, "role": "member" }
  ]
}
----
NOTE: `roles` contains the *system ACL role* (e.g. `user`), not the membership role.
The membership role (`owner`/`member`) is per-team and stored in the `teams` array.

`roles` is always kept in sync with the role for the active `team`.
`teams` is the authoritative list of all memberships.
Custom providers can use a different identifier type — see xref:membership-providers.adoc[Custom Membership Providers].

=== Adding a user to a second team

Inviting an existing user to a different team adds a new entry to their `teams` array without requiring account reactivation (the user is already active):

[source,http]
----
POST /auth/invite
Authorization: Bearer <owner-or-admin-token>
Content-Type: application/json

{ "email": "alice@example.com", "role": "member" }
----

* If the user already belongs to the caller's team → `409 Conflict`.
* If the user is new → standard invite flow (see xref:team-invitations.adoc[Team Invitations]).
* If the user is active in another team → invitation sent; membership added when the user accepts; `201 Created`.

=== List team memberships

`GET /auth/teams` — requires authentication.

Returns all teams the current user belongs to. The entry whose `active` field is `true` matches the team encoded in the current JWT.

[source,http]
----
GET /auth/teams
Authorization: Bearer <token>
----

[source,json]
----
[
  { "id": {"$oid": "64a1b2c3d4e5f6a7b8c9d0e1"}, "name": "Acme Corp",  "role": "owner",  "active": true  },
  { "id": {"$oid": "64a1b2c3d4e5f6a7b8c9d0e2"}, "name": "Other Corp", "role": "member", "active": false }
]
----

The `id` field is serialized as MongoDB extended JSON, preserving the native BSON type.
For ObjectId team identifiers (the default) it is `{"$oid": "..."}`.
Custom providers that use string or integer team IDs will produce a plain JSON string or number.

=== Switch active team

`POST /auth/switch-team` — requires authentication.

Verifies that the user belongs to the requested team, issues a new JWT with the correct role for that team, and updates the auth cookie.
The browser (or client) does not need to re-enter credentials.

[source,http]
----
POST /auth/switch-team
Authorization: Bearer <token>
Content-Type: application/json

{ "teamId": {"$oid": "64a1b2c3d4e5f6a7b8c9d0e2"} }
----

The `teamId` value must be in MongoDB extended JSON format, matching the `id` field returned by `GET /auth/teams`.

On success (`200 OK`):

* The response body confirms the new active team and role.
* The reissued JWT is delivered per the optional `delivery` query parameter — `cookie` (default; `Set-Cookie` carries the new JWT) or `body` (since 9.6.0; `access_token`/`token_type`/`expires_in` in the response body plus the `Auth-Token` header, no cookie).

[source,json]
----
{ "team": { "_id": {"$oid": "64a1b2c3d4e5f6a7b8c9d0e2"}, "role": "owner" }
----

==== Error responses

|===
| Status | Reason

| `400` | Missing or blank `teamId`
| `401` | Not authenticated
| `403` | User does not belong to the requested team, or account not active
| `404` | User document not found
|===

=== Typical frontend flow

[source,typescript]
----
// 1. Load team list on app init
authService.getTeams()  // GET /auth/teams
  .subscribe(teams => this.teams.set(teams));

// 2. User picks a different team
authService.switchTeam(teamId)  // POST /auth/switch-team
  .pipe(switchMap(() => authService.checkSession()))
  .subscribe(() => router.navigate(['/time']));
----

After `switchTeam`, call `checkSession()` (`GET /token`) so the app state reflects the new team and role.

=== ACL considerations

No manual ACL entries are needed for `/auth/teams` or `/auth/switch-team`.
Both endpoints register their own allow rules via `ACLRegistry` at startup.
They are disabled when `accountsConfig.membership-endpoints-enabled: false`.

See xref:membership-providers.adoc[Custom Membership Providers] to replace the built-in storage strategy.
