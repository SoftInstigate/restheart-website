Every Sophia instance exposes an link:https://modelcontextprotocol.io[MCP-compatible] server, allowing any AI client to query its knowledge base directly — without going through the chat UI or the REST API.

== What the MCP Server Exposes

Sophia organises its knowledge into *Contexts*. Each context defines an isolated knowledge domain with its own:

- *documents* — files ingested into the knowledge base, tagged for access control
- *system prompt template* — how the AI frames its answers
- *RAG options* — how many documents to retrieve and inject into the prompt
- *MCP description* — the text shown to AI clients when they list available tools

Each context is accessible at its own MCP endpoint: `<base-url>/mcp/<context-id>/`. When an AI client connects to that endpoint, it can use the Sophia tools to search and retrieve content *scoped to that context only*. A client connected to the `restheart` context cannot read documents that belong to the `cloud` context, and vice versa.

Within a context, documents are further organised by *tags*. Pass the optional `tags` parameter to `sophia_search` to narrow retrieval to documents carrying specific tags. Use `sophia_list_tags` to discover which tags are available.

For more on contexts, see the link:/docs/cloud/sophia/administrator-guide#_context_management[Administrator Guide].

== SoftInstigate Public MCP Servers

SoftInstigate runs two public Sophia instances as live examples — no account or token required:

[cols="1,2,2", options="header"]
|===
| Context | Url | Knowledge base
| `restheart` | `https://sophia-api.restheart.com/mcp/restheart/` | RESTHeart documentation, plugin API, configuration guides
| `cloud` | `https://sophia-api.restheart.com/mcp/cloud/` | RESTHeart Cloud managed service docs
|===

These are fully functional Sophia deployments managed by SoftInstigate. You can use them directly in your AI client.

To connect immediately with Claude Desktop, open *Settings → Connectors → Add custom connector* and paste the URL.

For Claude Code, run:

[source,bash]
----
claude mcp add --transport http sophia-restheart https://sophia-api.restheart.com/mcp/restheart
claude mcp add --transport http sophia-cloud https://sophia-api.restheart.com/mcp/cloud
----

For private contexts, add the API token via the `--header` flag:

[source,bash]
----
claude mcp add --transport http sophia-restheart https://<your-sophia>/mcp/<context>/ \
  --header "Authorization: Bearer <your-token>"
----

== Clients with Streamable HTTP transport

Add this to the MCP settings:

[source,json]
----
{
  "sophia": {
    "type": "http",
    "url": "https://<your-sophia>/mcp/<context>/"
  }
}
----

== Clients with stdio transport

Some clients (VS Code, Zed) spawn MCP servers as local processes and communicate over stdio. Use link:https://www.npmjs.com/package/mcp-remote[`mcp-remote`] as a bridge. It requires link:https://nodejs.org[Node.js >= 18] and is downloaded automatically by `npx` on first run.

Example configuration snippet:

[source,json]
----
{
    "sophia": {
      "command": "npx",
      "args": ["mcp-remote", "https://<your-sophia>/mcp/<context>/"]
    }
}
----

== Private contexts

Private contexts require authentication. A context is private when its `private` field is `true` in the admin panel; access requires a JWT containing the context id in the `contexts` claim. Sophia supports three authentication paths.

=== Via OAuth (recommended for Claude Desktop)

Claude Desktop supports OAuth natively and can re-authenticate transparently when its JWT expires.

. In Claude Desktop open *Settings → Connectors → Add custom connector*
. Paste the context URL (e.g. `https://your-sophia/mcp/support/`)
. Set *OAuth Client ID* to a value provisioned by your administrator (e.g. `claude_desktop`)
. Sophia's login page opens in the browser; complete the flow to authorise the connector

Sophia exposes the standard OAuth metadata endpoints under `/.well-known/oauth-authorization-server` and `/.well-known/oauth-protected-resource`. The grant used after the initial user authorisation is `client_credentials`, so subsequent re-authentications are silent.

=== Via API token (recommended for headless clients)

Issue a long-lived API token from the admin panel — see link:/docs/cloud/sophia/administrator-guide#_api_token_management[API Token Management] — and pass it as an `Authorization: Bearer` header. Tokens are revocable.

[source,json]
----
{
  "sophia": {
    "type": "http",
    "url": "https://your-sophia/mcp/{context}/",
    "headers": {
      "Authorization": "Bearer <your-token>"
    }
  }
}
----

=== Via `?token=` query parameter (clients without custom headers)

Some clients (notably Claude Mobile and certain web wrappers) cannot attach custom headers. Sophia accepts the JWT as a `?token=<jwt>` query parameter that is internally promoted to `Authorization: Bearer <jwt>`:

[source]
----
https://your-sophia/mcp/{context}/?token=<your-token>
----

This is the same JWT issued by the admin panel — there is no separate token type.

Example configuration snippet:

[source,json]
----
{
  "sophia": {
    "type": "http",
    "url": "https://<your-sophia>/mcp/<context>/",
    "headers": {
      "Authorization": "Bearer <your-token>"
    }
  }
}
----

For clients that support stdio only:

[source,json]
----
{
    "sophia": {
      "command": "npx",
      "args": [
        "mcp-remote",
        "https://<your-sophia>/mcp/<context>/",
        "--header", "Authorization: Bearer <your-token>" ]
    }
}
----

== Available Tools

[cols="1,3", options="header"]
|===
| Tool | Description
| `sophia_search` | Semantic search over the knowledge base. Returns relevant text segments with `Source:` filename and `Path:` directory for context expansion. Supports an optional `compact` flag for source-only previews (~10× cheaper).
| `sophia_get_file` | Retrieves a paginated page of a document by filename. Use after `sophia_search` to read beyond the matched chunk; pass `page` to navigate.
| `sophia_list_files` | Lists all accessible documents with tags and chunk count. Accepts an optional `path` parameter to filter by directory.
| `sophia_list_tags` | Lists the tags available in the knowledge base, filtered by the active context and credentials.
| `sophia_list_paths` | Lists immediate subdirectories under a given path prefix, like `ls`. Without arguments, returns top-level directories.
| `sophia_render_prompt` | Builds a fully interpolated system prompt with RAG context injected, ready to send to any LLM as a `system` + `user` message pair.
| `sophia_api` | Returns the Sophia REST API reference. Requires authentication — reserved for service administrators.
|===

NOTE: `render_input` / `render_output` (collaborative artifacts) and the `sophia_context_*` tools (agentic context management) are *not* exposed via MCP — they are internal tools used by Sophia's own agentic loop and post-answer synthetic calls.
